# Dead-End Protocol

## How to Recognize a Dead End

You're at a dead end when:
- 3+ attempts at the same approach have failed
- Results aren't improving despite changes
- You can't explain WHY the approach should work
- You're spending more time debugging than learning
- The same errors keep recurring

## What to Do

### Step 1: STOP
Stop whatever you're doing. Don't try "one more thing."

### Step 2: Document
Write in the vault:
- What was the approach?
- What specifically failed?
- What did I learn from the failure?
- Why do I think it failed? (honest assessment)

### Step 3: Step Back
Ask big-picture questions:
- Am I solving the right PROBLEM?
- Is my understanding of the domain correct?
- Am I missing a fundamental concept?
- Is there prior work I haven't read?

### Step 4: Research
- Search for how others solved similar problems
- Read papers in adjacent fields
- Look at different tools/frameworks/approaches
- Talk to the user — they might have domain knowledge you lack

### Step 5: Pivot or Reframe

**Option A: Change approach entirely**
Different algorithm, different data, different tool. Start fresh on the implementation but keep the knowledge.

**Option B: Reframe the problem**
Maybe the goal itself needs adjustment. "Make the strategy profitable" might need to become "understand why the strategy loses" first.

**Option C: Simplify**
Solve an easier version of the problem first. Build up from there.
- Can't make a complex strategy work? Start with the simplest possible version.
- Can't process all the data? Start with one pair, one timeframe.

### Step 6: Update the Plan
Write the new direction in:
- Vault (document the pivot and why)
- IDENTITY.md (update approach and methodology)
- CLAUDE.md (if there's a lesson about dead-end detection)

## The Key Insight

**Dead ends are data.** They tell you what DOESN'T work, which narrows the search space. A well-documented dead end is almost as valuable as a success because it prevents future iterations from repeating it.

When documenting a dead end, classify what went wrong:
- **Memory error** — was the dead end caused by stale or wrong vault knowledge?
- **Reasoning error** — did a flawed hypothesis lead here?
- **Planning error** — was the approach fundamentally wrong from the start?
- **Action error** — was the right idea executed incorrectly?

This classification helps write a targeted self-evolution rule that prevents the same CLASS of dead end.

## NEVER

- Keep trying the same thing hoping for different results
- Blame the tools, the data, or the market without evidence
- Give up entirely — pivot is not giving up, it's redirecting
- Delete the failed work without documenting WHY it failed
